# Provenance Guard

A backend service a creative-sharing platform can plug in to estimate whether submitted text was
written by a human or generated by AI, express that estimate **with honest uncertainty**, show
readers a plain-language transparency label, and let creators appeal.

Guiding principle: **on a writing platform, wrongly labeling a real author's work as AI is the worst
outcome.** The whole system is tuned to be slow to accuse and quick to say "uncertain."

> Built with Flask. Detection uses a 3-signal weighted ensemble (semantic + structural + lexical).
> Signal 1 calls **Groq `llama-3.3-70b-versatile`** when `GROQ_API_KEY` is set, and falls back to a
> deterministic offline heuristic when it isn't — so the system runs and is testable anywhere, and
> uses the real model the moment a key is present (no code change).

## Quick start

```bash
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt

cp .env.example .env               # then add your GROQ_API_KEY (optional; offline fallback works without it)
python app.py                      # serves on http://localhost:5000
```

Submit a piece of text:

```bash
curl -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "The sun dipped below the horizon...", "creator_id": "test-user-1"}' | python -m json.tool
```

### Endpoints

| Method & path | Purpose |
| --- | --- |
| `POST /submit` | Classify text → attribution, confidence, transparency label, individual signal scores. Rate limited. |
| `POST /appeal` | Contest a classification (`content_id` + `creator_reasoning`) → status `under_review`, logged. |
| `GET /log` | Recent structured audit entries (JSON). `?limit=N` optional. |
| `GET /analytics` | Platform metrics: verdict mix, appeal rate, avg confidence (stretch). |
| `GET /health` | Liveness. |

## Architecture overview — the path a submission takes

1. **`POST /submit`** receives `{ text, creator_id }`, passes the **rate limiter**, validates the
   input, and mints a unique `content_id`.
2. **Signal 1 — LLM semantic classifier** (`signals.llm_semantic_signal`) reads the text's
   voice/register and returns `semantic` ∈ [0,1] (probability AI).
3. **Signal 2 — stylometric heuristics** (`stylometry.stylometry_signal`) measures the text's
   structure and returns `structural` ∈ [0,1].
4. **Signal 3 — lexical fingerprint** (`signals.lexical_signal`) measures surface AI-cliché vs.
   casual-human phrasing and returns `lexical` ∈ [0,1].
5. **Fusion** (`scoring.score`) combines them into `combined_p_ai` (weighted mean) and a separate
   `confidence` value that **drops when the signals disagree**.
6. **Attribution + label** (`labels.label_for`): `combined_p_ai` → `likely_ai` / `uncertain` /
   `likely_human`; that verdict **plus** the confidence selects one of three plain-language labels.
7. **Audit log** (`audit.append_entry`) writes a structured JSON record.
8. **Response** returns the verdict, confidence, label, and every individual signal score.

**Appeal path:** `POST /appeal` looks up the original decision by `content_id`, flips its status to
`under_review`, writes a new audit entry carrying the creator's reasoning **alongside** the original
scores, and returns a confirmation for a human reviewer.

See `planning.md` for the full ASCII architecture diagram of both flows.

## Detection signals

The pipeline uses **three distinct signals**, each reading a different property of the text so they
fail in different directions. (Two are required; the third is the Ensemble stretch — see below.)

| Signal | Type | What it measures | Why it differs human↔AI | What it **misses** (blind spot) |
| --- | --- | --- | --- | --- |
| **1. LLM semantic** (Groq) | Semantic / voice | Whether the prose carries a consistent human voice and lived specificity vs. a smooth, hedged, formal "AI register" | Humans write idiosyncratically; models trend toward balanced, safe, evenly-hedged prose | Mislabels **polished/formal/non-native-English** human writing as AI; can be fooled by deliberately roughened AI; non-deterministic and not fully auditable |
| **2. Stylometry** | Structural | Sentence-length **burstiness** (coefficient of variation), **mean sentence length**, **expressive-punctuation density** | AI text is statistically more *uniform* (low variance, longer even sentences, less varied punctuation); human text is more variable | **Blind to meaning.** A repetitive simple-vocab human poem looks uniform → false positive; a varied AI essay looks bursty → false negative. Needs enough text. |
| **3. Lexical fingerprint** | Lexical / surface | Density of AI cliché/buzzword phrases vs. casual human markers and slang | AI over-uses a recognizable register ("paradigm", "leverage", "in today's digital landscape"); humans use contractions, slang, fragments | **Keyword-based and gameable.** Misses paraphrased AI with no clichés; can ding a human who happens to write formally. |

**Why this combination:** one signal reads *meaning/voice*, one reads *structure*, one reads
*surface lexicon*. Because they're independent, their **agreement** corroborates a verdict and their
**disagreement** is the clearest honest evidence of "we don't actually know" — which is fed directly
into the confidence score.

### How they're combined (ensemble)

```
combined_p_ai = 0.55·semantic + 0.25·structural + 0.20·lexical      (weights sum to 1)
```

Semantic carries the most weight (most reliable); lexical the least (noisiest). Each individual
signal score is returned in `/submit` and written to the audit log, so the ensemble is transparent.

## Confidence scoring with uncertainty

The system returns **two numbers** because they answer different questions:

- **`combined_p_ai`** — *which way* the evidence leans (AI vs. human).
- **`confidence`** — *how sure* we are in that lean.

### Attribution thresholds (deliberately asymmetric)

| `combined_p_ai` | Attribution |
| --- | --- |
| ≥ 0.70 | `likely_ai` |
| 0.40 – 0.6999 | `uncertain` |
| < 0.40 | `likely_human` |

The "AI" cutoff is set high (0.70) with a **wide uncertain band** beneath it. This is the
false-positive-aversion knob: it takes strong evidence to call something AI; anything ambiguous
defaults to "uncertain" rather than an accusation.

### Confidence formula

```
extremity  = 2 · |combined_p_ai − 0.5|         # 0 at the midpoint, 1 at the extremes
agreement  = 1 − (max signal − min signal)     # 1 when signals agree, →0 when they diverge
confidence = clamp(0.6 · extremity + 0.4 · agreement, 0, 1)
```

A verdict is only confident when it is **both** far from the middle **and** backed by agreeing
signals. The label is then **gated by confidence**: a `likely_ai`/`likely_human` lean that scores
below **0.70 confidence collapses to the Uncertain label**, so a reader is never shown a confident
verdict the system can't stand behind.

### How I validated the scores are meaningful

I ran a battery of deliberately chosen inputs spanning the spectrum and confirmed the scores move as
intuition demands and that **all three label variants are reachable**. Highlights:

| Input | semantic | structural | lexical | `combined_p_ai` | `confidence` | Label shown |
| --- | --- | --- | --- | --- | --- | --- |
| Long, uniform, clichéd Ai essay | 0.80 | 0.81 | 0.90 | **0.823** | **0.748** | **Likely AI-generated** |
| Casual human review ("ramen…") | 0.22 | 0.17 | 0.15 | **0.195** | **0.736** | **Likely human-written** |
| Formal human (monetary policy) | 0.70 | 0.81 | 0.50 | **0.688** | **0.501** | **Attribution uncertain** |
| Short clichéd AI (not structurally uniform) | 0.77 | 0.52 | 0.90 | 0.73 | 0.53 | Attribution uncertain |
| Lightly-edited / hybrid AI | 0.44 | 0.24 | 0.50 | 0.40 | 0.41 | Attribution uncertain |

Two takeaways this validates:
- **Confidence varies meaningfully, not constantly** — compare the high-confidence AI case (**0.748**)
  with the formal-human case (**0.501**). Same system, very different confidence and a different label.
- **The false-positive asymmetry works end-to-end.** The formal human essay *leans AI*
  (`combined_p_ai` 0.688) but is shown **"Attribution uncertain," not an accusation**, because the
  signals partly disagree and confidence (0.501) fails the gate. The short clichéd AI text is
  similarly held at "uncertain" because only the lexical/semantic signals fire while structure
  doesn't — high confidence requires the signals to *agree*.

### Two example submissions with noticeably different confidence

**High-confidence example** (clearly AI):
```json
{ "attribution": "likely_ai", "combined_p_ai": 0.823, "confidence": 0.748,
  "signals": { "semantic": 0.80, "structural": 0.81, "lexical": 0.90 },
  "label": { "headline": "Likely AI-generated", "level": "High confidence" } }
```

**Lower-confidence example** (formal human writing — leans AI but not confidently):
```json
{ "attribution": "uncertain", "combined_p_ai": 0.688, "confidence": 0.501,
  "signals": { "semantic": 0.70, "structural": 0.81, "lexical": 0.50 },
  "label": { "headline": "Attribution uncertain", "level": "Low confidence — inconclusive" } }
```

## Transparency label — the three variants (verbatim)

Plain language, no jargon, false-positive-aware. The label **changes with confidence**, not just the
number. Exact text:

| Variant | `headline` | `level` | `body` (verbatim) |
| --- | --- | --- | --- |
| **High-confidence AI** | `Likely AI-generated` | `High confidence` | "Our automated checks strongly indicate this text was generated by AI. Two independent checks — one that reads the writing's style and voice, and one that measures its structure and rhythm — both pointed the same way. This is a confident estimate, not proof. If you're the creator and you wrote this yourself, you can dispute it and a person will review it." |
| **High-confidence human** | `Likely human-written` | `High confidence` | "Our automated checks strongly indicate a person wrote this text. Both its voice and its structure look characteristic of human writing. This is a confident estimate, not a guarantee." |
| **Uncertain** | `Attribution uncertain` | `Low confidence — inconclusive` | "We can't tell with confidence whether this text was written by a person or generated by AI — our checks either disagreed or landed in the middle. AI detection is imperfect, and formal, technical, or non-native-English writing is easy to misread as AI. We'd rather flag the uncertainty than guess. If you're the creator, you can add context through an appeal." |

A low-confidence AI/human lean deliberately renders the **Uncertain** variant, so a reader never sees
a confident accusation the system can't back.

## Appeals workflow

- **Who:** the creator, referencing the `content_id` they received. (This prototype does not enforce
  hard auth on `creator_id`; a real deployment would verify identity — noted as a limitation.)
- **What they send:** `{ content_id, creator_reasoning }`.
- **What the system does:** looks up the original decision (404 if unknown), flips its status
  `classified → under_review`, appends a **new** audit entry of type `appeal` carrying the reasoning
  plus the original attribution, confidence, and all three signal scores, and returns a confirmation.
  No automated re-classification (per spec).
- **Reviewer view:** the appeal queue is every record with `status == under_review`; the reviewer
  sees the original scores and the creator's reasoning side by side.

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-ID", "creator_reasoning": "I wrote this myself; I am a non-native English speaker and my register is formal."}'
# -> { "content_id": "...", "status": "under_review", "message": "Appeal received. This content is now under human review." }
```

## Rate limiting

Applied to `POST /submit` (read endpoints are unlimited):

```
@limiter.limit("10 per minute;100 per day")
```

**Reasoning (not arbitrary):** on a writing platform a real creator submits their own work
occasionally — a handful of pieces in a sitting at most, including revisions and resubmissions. **10
per minute** comfortably covers a human iterating on one piece while stopping a script from hammering
the cost-bearing LLM signal. **100 per day** caps sustained abuse and runaway LLM spend from a single
source over time — a legitimate individual creator essentially never approaches 100 submissions in a
day, but an automated flood would. The two windows defend against both burst abuse (per-minute) and
slow-drip abuse (per-day).

### Evidence — limit triggers (12 rapid requests)

```
12 rapid POST /submit -> [200, 200, 200, 200, 200, 200, 200, 200, 200, 200, 429, 429]
200s: 10   |   429s: 2

429 body: { "error": "rate limit exceeded", "limit": "10 per 1 minute",
            "message": "Too many submissions. Please slow down and try again shortly." }
```

## Stretch features

### Ensemble detection (built)
The deployed pipeline is a **3-signal weighted ensemble** — semantic `0.55`, structural `0.25`,
lexical `0.20`. **Conflict resolution:** the weighted mean sets the verdict; disagreement is measured
as the spread (max − min) across the three signal scores and lowers confidence via `agreement = 1 −
spread`, and any verdict that fails the 0.70 confidence gate is shown as "uncertain." Every `/submit`
response and audit entry shows the individual signal scores **and** the ensemble result, plus an
`ensemble` block with the weights and the measured agreement/spread.

### Analytics dashboard (built)
`GET /analytics` returns, computed from the audit log: the **detection pattern** (counts and ratios
of likely_ai / uncertain / likely_human), the **appeal rate**, **average confidence**, and the
**low-confidence share**. Example:

```json
{ "total_classifications": 3, "verdict_ratios": { "likely_ai": 0.333, "uncertain": 0.333, "likely_human": 0.333 },
  "total_appeals": 1, "appeal_rate": 0.333, "average_confidence": 0.662, "low_confidence_share": 0.333 }
```

## Known limitations

- **Repetitive, simple-vocabulary poetry (structural false positive).** A human poem built on
  refrains, short even-length lines, and plain words produces *low sentence-length variance* and a
  short, uniform structure — exactly what the **stylometry** signal reads as AI-like. Because that
  signal is blind to meaning, it can't tell deliberate poetic repetition from machine uniformity, so
  it pushes such poems toward "AI." The wide uncertain band and low signal weight (0.25) soften this,
  but heavily-patterned verse genuinely sits near the boundary and may be shown "uncertain" when a
  human clearly wrote it.
- **Formal / non-native-English prose.** Formal register inflates both the **semantic** (reads as
  "AI register") and **structural** (long, even sentences) signals, so this writing leans AI on
  `combined_p_ai`. The confidence gate is what protects it — it's shown "uncertain," not accused —
  but the leaning itself is a real weakness of these signals.
- **Offline mode reduces signal independence.** Without a `GROQ_API_KEY`, Signal 1 uses a lexical/
  register heuristic that shares some surface cues with Signal 3, so the two are less independent than
  when the real semantic model is used. With a key, Signal 1 is genuinely semantic and the ensemble
  is stronger.
- **No perfect detection.** AI detection is an unsolved problem; this system is designed to be honest
  about uncertainty and to give creators an appeal, not to be a definitive judge.

## Spec reflection

**One way the spec helped:** writing the confidence model in `planning.md` *before* coding — defining
`combined_p_ai` (which way) and `confidence` (how sure) as two separate numbers, with the
confidence-gate rule that collapses low-confidence leans to "uncertain" — meant the false-positive
protection was an architectural decision, not an afterthought. When implementation revealed that
formal human writing leans AI, the system already did the right thing (showed "uncertain") with no
redesign.

**One way the implementation diverged from the plan:** `planning.md` specced **type-token ratio
(TTR)** as the second stylometric metric. In testing, TTR proved badly **length-biased** — on the
short submissions typical of a writing platform, almost every text scores high vocabulary diversity,
so TTR read "human" for AI and human text alike and added no separation (`uniformity_ttr` was pinned
near 0 for nearly everything). I replaced it with **average sentence length**, which discriminates
reliably on short text (AI tends to longer, even sentences). TTR is still computed and reported as an
informational metric, but it is no longer part of the score. A second, smaller divergence: I had to
**recalibrate the offline semantic stand-in** downward after discovering the high-confidence *human*
label was unreachable — the proxy floored too high for casual human voice, so confidence couldn't
clear the gate.

## AI usage

This project was built with AI assistance (Claude). Specific instances:

1. **Stylometry function from the spec.** I directed the AI to generate `stylometry_signal` from the
   planning.md signal description (burstiness, TTR, punctuation) returning a single 0–1 score. The
   output implemented TTR as specced. On testing against the four calibration inputs I found TTR was
   length-biased and contributed no separation, so I **overrode the spec and the generated code**,
   replacing TTR with average sentence length and re-deriving the sub-metric weights
   (0.45/0.30/0.25). I kept TTR as an informational field for transparency.

2. **Confidence formula and label gating.** I directed the AI to implement the fusion + confidence
   formula and the threshold table exactly as written in planning.md §2. The generated scoring
   matched the spec — but running the real pipeline revealed the **high-confidence AI label was
   unreachable** (a strong semantic signal disagreeing with a middling structural signal capped
   confidence ~0.6). Rather than just lower the gate (which would have weakened false-positive
   protection), I **decided differently**: I kept the 0.70 gate and recalibrated the signals so that
   high confidence genuinely requires the signals to *agree* — and verified a longer, uniform AI text
   then reaches the confident label while short clichéd-but-irregular AI honestly stays "uncertain."

3. **Ensemble generalization.** I directed the AI to extend the 2-signal fusion to a 3-signal
   weighted ensemble for the stretch feature. I **revised** its first cut so that the agreement term
   generalized cleanly (`agreement = 1 − spread`, which reduces to `1 − |a − b|` for two signals),
   keeping the required 2-signal behavior identical rather than introducing a second code path.
